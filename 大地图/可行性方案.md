# 大地图功能 — 可行性方案

> 对应《需求分析.md》：45° 视角大地图、超大网格、分层、缩放平移、性能优化。  
> 技术栈：**Cocos Creator 3.8.x**（与当前工程版本一致）。

---

## 一、总体结论

| 需求 | 可行性 | 成熟做法概要 |
|------|--------|----------------|
| 5000×5000 格子 | **可行** | 数据可表示全图，**显示层仅加载视口附近分块（Chunk Streaming）** |
| 地图分层 | **可行** | **逻辑分层 + 渲染分层**（多节点树 / Tiled 多 Layer / 多摄像机） |
| 放大缩小、移动 | **可行** | **正交摄像机**调节视口 + 位置边界限制；或地图根节点统一缩放与位移 |
| 性能优化 | **可控** | **视口裁剪、分块、图集合批、对象池、LOD、寻路与碰撞与渲染解耦** |

**要点**：全图约 2500 万个格子，**不能**为每格创建独立节点/Sprite；业界通用做法是 **Chunk 流式加载 + 只渲染可见区域**。

## 二、超大尺寸地图（需求 1）

### 2.1 成熟数据与工具链

1. **Tiled Map Editor + TMX / JSON**
   - 大地图常见做法：按区域导出**多块**（如每块 256×256 或 512×512 格），或使用 Tiled 的 **Infinite map / 分块导出**。
   - Cocos 提供 **`TiledMap`**（TMX），适合中等规模与多层编辑；**单一超大 TMX** 加载与解析成本高，工业界普遍 **拆成多 chunk 文件**，而不是一个 5000×5000 的单文件。

2. **纯数据 + 自研加载（灵活）**
   - 用 JSON / 二进制 / SQLite 等按 `(chunkX, chunkY)` 存 `tileId` 或 `layer + tileId`。
   - 与 Roguelike、2D 开放世界常用的 **chunk 流式** 模型一致，与具体引擎解耦，在 Cocos 中用 TypeScript 管理即可。

### 2.2 推荐架构

- **逻辑**：仍可视为 5000×5000 的索引空间；稀疏区域可用 **稀疏结构**（只存有内容的 chunk）。
- **运行时**：设定 `chunkSize`（如 32 / 64 / 128），仅加载摄像机周围 **N×N** 个 chunk；跨越边界时 **卸载远端 chunk、加载新进 chunk**。

节点数量上限 ≈ **可见 chunk 数 × 单 chunk 内展示量**，而非 2500 万。

## 三、地图分层（需求 2）

### 3.1 逻辑与显示分离

| 层类型 | 典型内容 | Cocos 中常见实现 |
|--------|----------|------------------|
| 地面 / 地表 | 草地、路面等 | 子节点如 `Layer_Ground`，或 Tiled 的 ground layer |
| 碰撞 / 阻挡 | 不可走、占位 | **以数据为主**（二维数组、位图、导航网格）；可不渲染或仅调试用极简显示 |
| 建筑 | 房屋、设施 | 子节点 `Layer_Buildings`；等距场景常用 **按脚底 Y 或排序值** 决定前后遮挡 |
| 特效 | 粒子、闪光等 | 子节点 `Layer_FX`，保证渲染顺序在最后或单独排序 |

### 3.2 实现方式（可组合）

- **节点树分层**：根节点下挂多个 Layer 节点，每层内部再按 chunk 管理子节点。
- **Tiled 多 Layer**：一层一类内容；大地图仍建议 **多文件 chunk**，每 chunk 一个 TMX 或合并 layer 数据后自研解析。
- **碰撞层**：优先 **不画或少画**，单独供寻路 / 范围检测，减少 DrawCall。

## 四、缩放与移动（需求 3）

### 4.1 推荐：摄像机驱动（2D / 伪 45° 大地图最常用）

- **平移**：调整 `Camera` 节点 `worldPosition`（或移动地图根节点，二者择一并统一坐标系）。
- **缩放**：修改正交摄像机 **`orthoHeight`**（或缩放挂载整块地图的父节点；注意与 UI 分层，避免误缩放界面）。
- **边界**：将地图宽高换算为世界单位后，**夹紧**相机或地图位移，避免露出空白边界。
- **输入**：鼠标拖拽 + 滚轮；移动端单指拖拽 + 双指捏合，使用 Cocos 触摸事件即可。

### 4.2 补充

- 若仅为 **UI 小地图**，可考虑 `ScrollView` + `content` 缩放；**主场景大地图**优先 **World + Camera**，更易控性能与坐标。

## 五、性能优化（需求 4）

### 5.1 与引擎无关的成熟手段

1. **视口 / Chunk 裁剪**：只创建视口内（可加一圈 margin）的 chunk 与瓦片。
2. **对象池**：chunk 卸载时回收节点，避免频繁 `instantiate` / `destroy`。
3. **图集（Atlas）**：瓦片与建筑贴图打 **Texture Packer / Cocos 自动图集**，减少材质切换与 DrawCall。
4. **同图集、同材质**：尽量批量使用相同材质路径，利于合批。
5. **建筑 LOD**：远景简化或占位图，近景再换精细资源（策略、模拟类常用）。
6. **寻路 / 重逻辑**：可对 chunk 做 **粗网格** 或 **分层分块寻路**；Web 平台可将重计算放入 **Web Worker**，减轻主线程卡顿。

### 5.2 Cocos Creator 3.8 侧关注点

- 规则瓦片可优先考虑内置 **TileMap** 与合批能力。
- 关注 **DrawCall、节点数、纹理切换**；用编辑器 Profiler 验证优化效果。
- 文档与版本差异以 [Cocos 官方文档](https://docs.cocos.com/) 当前版本为准（动态合批、GPU Instancing 等）。

## 六、推荐落地步骤

1. **定坐标系**：格子 `(gx, gy)` ↔ 世界坐标（等距常用：由 `halfTileWidth`、`halfTileHeight` 推导 `x,y`）。
2. **选数据源**：强编辑流程 → **Tiled 分块**；程序生成或超大存档 → **自研 chunk 格式**。
3. **实现 ChunkLoader**：根据相机中心计算 chunk 集合 → 异步加载 / 卸载 → 对象池复用节点。
4. **搭建四层结构**：地面、建筑、特效为显示层；碰撞为数据层（可选调试图层）。
5. **CameraController**：平移、缩放、边界、输入绑定。
6. **压测与迭代**：Profiler 定位瓶颈，再决定是否上 LOD、Worker、更激进的数据稀疏化。

---

## 七、风险与边界

- **5000×5000 全量内存**：若每格多字节存储，内存仍可能过大 → 采用 **稀疏存储**、**仅加载周边 chunk 数据**、或磁盘流式。
- **避免单文件巨型 TMX**：应 **分 chunk 导出与加载**。
- **若未来改为真 3D 场景**：分块与流式思路不变，渲染侧改为 **网格 / 地形分块 + 视锥裁剪** 即可。
