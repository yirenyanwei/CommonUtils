# 战斗代码类说明

本文档说明 `assets/game/scripts/battle/` 下第一版战斗框架代码的类、接口和模块职责。当前代码目标是先完成“单机逻辑运行 + View 分离”的最小闭环：本地固定 tick 推进、右到左刷怪、放置塔、投射物命中、事件驱动表现。

如果想按运行顺序理解代码，优先阅读 `文档/战斗框架设计与执行流程.md`；本文档更偏类职责索引。

## 总体调用链

运行入口是 `TestGame`，但它只负责接入 Cocos 生命周期。正式战斗启动、命令输入和 View 适配已经拆到 `BattleSceneController` 与 `BattleApplication`。

```text
TestGame
  -> BattleSceneController
      -> BattleApplication
          -> BattleCore
              -> BattleCommandQueue
              -> BattleDirector
                  -> CommandSystem
                  -> WaveSystem
                  -> UnitSystem
                  -> AutoAttackSystem
                  -> ProjectileSystem
                  -> BattleResultSystem
              -> BattleEventBus
      -> BattleViewAdapter
```

核心约束：

- `application/battle_application.ts` 和 `core`、`system`、`command`、`event`、`unit`、`ability`、`modifier`、`wave`、`resolver`、`random`、`replay`、`data` 不依赖 Cocos。
- `application/battle_scene_controller.ts` 和 `view` 目录允许依赖 `cc`，只负责场景接入和显示，不参与命中、伤害、胜负判断。
- 逻辑对象通过事件通知 View，View 不反向修改逻辑状态。

## Application 模块

### `BattleApplication`

文件：`battle/application/battle_application.ts`

职责：

- 作为正式的战斗用例层。
- 创建并持有 `BattleCore` 与 `BattleRunner`。
- 对外提供 `startBattle()`、`update()`、`placeTower()`、`castAbility()` 等用例方法。
- 统一维护命令序号，避免入口层直接拼装命令细节。
- 不依赖 Cocos，后续无 UI 测试或服务器复盘也可以复用这一层。

关键关系：

- UI、场景、网络层不建议直接操作 `BattleCore`，优先通过 `BattleApplication` 发起战斗行为。
- `BattleApplication.update()` 返回逻辑事件，由外部决定是否交给 View。

### `BattleSceneController`

文件：`battle/application/battle_scene_controller.ts`

职责：

- 作为 Cocos 场景侧的正式控制器。
- 创建 `BattleApplication`。
- 创建 `BattleViewAdapter` 并绑定场景根节点。
- 提供 `startDemoBattle()` 启动当前示例战斗。
- 在 `update(deltaTime)` 中推进 `BattleApplication`，并把事件交给 `BattleViewAdapter`。

关键关系：

- 它可以依赖 `cc`，但不参与战斗裁决。
- 后续正式项目中，可以继续扩展为加载关卡配置、响应 UI 操作、连接战斗 HUD 的场景控制器。

## 入口示例

### `TestGame`

文件：`assets/game/scripts/test_game.ts`

职责：

- 作为 Cocos 场景里的测试入口组件。
- 在 `onLoad()` 中创建 `BattleSceneController`。
- 调用 `BattleSceneController.startDemoBattle()` 启动示例战斗。
- 在 `update(deltaTime)` 中把 Cocos 帧更新转发给 `BattleSceneController`。

设计说明：

- `TestGame` 不再直接创建 `BattleCore`、`BattleRunner`、`BattleViewAdapter`。
- `TestGame` 不再直接拼装放塔或释放技能命令。
- 它现在只是测试入口，正式业务职责已经下沉到 `BattleSceneController` 和 `BattleApplication`。

## Core 模块

### `BattleCore`

文件：`battle/core/battle_core.ts`

职责：

- 战斗逻辑对外的主入口。
- 持有 `BattleWorld`、`BattleCommandQueue`、`BattleDirector`。
- 提供 `enqueueCommand()` 接收外部命令。
- 提供 `step()` 推进一个逻辑 tick。
- 提供 `runUntilFinished()` 支持无 UI 一次性跑完整场战斗。

关键关系：

- 外部不应该直接操作大部分战斗流程，而是通过 `BattleCore` 输入命令。
- `BattleCore.step()` 每次会取出当前 tick 可执行命令，交给 `BattleDirector` 调度，然后返回本 tick 产生的事件。

### `BattleWorld`

文件：`battle/core/battle_world.ts`

职责：

- 一场战斗的完整运行时状态容器。
- 保存当前 tick、战斗结果、统一随机源、事件总线和各类 Manager。
- 持有 `UnitManager` 和 `ProjectileManager`。
- 负责发出战斗结束事件。

关键关系：

- `BattleWorld` 是核心状态层，不依赖 Cocos。
- `BattleDirector` 管理系统顺序，各个 `BattleSystem`、`Ability`、`UnitController` 都围绕 `BattleWorld` 读写逻辑状态。
- 单位和投射物的具体管理逻辑已经拆到 Manager 中，避免 `BattleWorld` 继续膨胀。

### `BattleDirector`

文件：`battle/core/battle_director.ts`

职责：

- 一场战斗每个 tick 的系统管理器。
- 按固定顺序执行 `BattleSystem` 列表。
- 创建 `BattleSystemContext`，把当前 tick 命令和跨系统状态传给系统。
- 当某个系统让战斗结束时，立即停止后续系统执行。

关键关系：

- 它不再承载具体战斗规则，只决定每 tick 谁先执行、谁后执行。
- 后续新增 Buff、光环、陷阱等逻辑时，优先新增系统并插入 `systems` 数组。

### `BattleRunner`

文件：`battle/core/battle_runner.ts`

职责：

- 把真实 `deltaTime` 转换成固定 tick 推进。
- 适配 Cocos 的 `update(deltaTime)`。
- 渲染帧率可以变化，但战斗逻辑按 `config.tickRate` 固定推进。

使用场景：

- Cocos 客户端运行战斗时使用 `BattleRunner.update(deltaTime)`。
- 无 UI 测试或服务器复盘时，可以直接使用 `BattleCore.step()` 或 `BattleCore.runUntilFinished()`。

## System 模块

### `BattleSystem`

文件：`battle/system/battle_system.ts`

职责：

- 定义所有战斗系统统一接口。
- 每个系统都有 `name` 和 `update(world, context)`。
- `BattleDirector` 只依赖这个接口执行系统管线。

### `BattleSystemContext`

文件：`battle/system/battle_system_context.ts`

职责：

- 保存单个 tick 内系统共享的上下文。
- 当前包含本 tick 命令列表和波次是否完成的状态。
- 后续可以加入调试信息、性能统计、当前阶段标记等跨系统数据。

### `CommandSystem`

文件：`battle/system/command_system.ts`

职责：

- 执行本 tick 的外部输入命令。
- 当前处理放塔和释放技能。
- 内部持有 `AbilityBook`，用于执行技能释放。

### `WaveSystem`

文件：`battle/system/wave_system.ts`

职责：

- 包装 `WaveScheduler`。
- 推进回合、波次和刷怪节奏。
- 将波次是否全部完成写入 `BattleSystemContext.isWaveComplete`。

### `UnitSystem`

文件：`battle/system/unit_system.ts`

职责：

- 更新单位行为。
- 当前主要调用 `UnitController` 处理敌人从右往左移动。
- 单位位置变化后发出 `UNIT_MOVED` 事件。

### `AutoAttackSystem`

文件：`battle/system/auto_attack_system.ts`

职责：

- 处理玩家阵营非主角单位的自动攻击。
- 当前主要服务防御塔。
- 自动攻击只创建投射物，伤害仍由投射物命中后结算。

### `ProjectileSystem`

文件：`battle/system/projectile_system.ts`

职责：

- 更新投射物飞行。
- 处理投射物命中、伤害结算和死亡事件。
- 当前内部调用 `world.projectileManager.update()`。

### `BattleResultSystem`

文件：`battle/system/battle_result_system.ts`

职责：

- 检查胜负条件。
- 主角死亡则失败。
- 超过最大 tick 则失败。
- 波次全部完成且敌人清空则胜利。

## Manager 模块

### `UnitManager`

文件：`battle/manager/unit_manager.ts`

职责：

- 管理所有战斗单位。
- 创建单位并发出 `UNIT_SPAWNED` 事件。
- 查询单位、存活单位、主角、敌人数量。
- 维护单位配置索引。
- 发出单位移动、受伤、死亡事件。
- 使用死亡 ID 集合避免同一个单位重复发出死亡事件。

关键关系：

- `CommandSystem` 和 `WaveScheduler` 通过它创建单位。
- `UnitSystem` 通过它查询存活单位和发出移动事件。
- `BattleResultSystem` 通过它查询主角和敌人数量。

### `ProjectileManager`

文件：`battle/manager/projectile_manager.ts`

职责：

- 管理所有投射物。
- 创建投射物并发出 `PROJECTILE_SPAWNED` 事件。
- 每 tick 更新投射物飞行。
- 判断投射物是否命中目标。
- 命中后调用 `DamageResolver` 结算伤害。
- 通过 `UnitManager` 发出受伤和死亡事件。

关键关系：

- `Ability` 和 `AutoAttackSystem` 通过它创建投射物。
- `ProjectileSystem` 每 tick 调用它更新投射物。
- `BattleWorld` 只持有它，不直接处理投射物细节。

## Data 模块

### `battle_config.ts`

文件：`battle/data/battle_config.ts`

职责：

- 定义战斗配置、单位配置、波次配置、开局参数、战斗结果等基础数据结构。
- 定义 `BattleUnitType`、`BattleTeam` 等枚举。
- 提供 `BattleVector` 和 `cloneVector()`。

主要类型：

- `BattleVector`：逻辑坐标。
- `BattleUnitType`：单位类型，包括主角、敌人、塔、友方。
- `BattleTeam`：阵营。
- `UnitStatsConfig`：血量、攻击、射程、攻击间隔、移动速度、投射物速度。
- `UnitConfig`：单位静态配置。
- `SpawnConfig`：单组刷怪配置。
- `WaveConfig`：波次配置。
- `RoundConfig`：回合配置。
- `BattleConfig`：整场战斗配置。
- `BattleStartParams`：开局参数。
- `BattleResult`：战斗结果。

### `createSampleBattleConfig`

文件：`battle/data/sample_battle_config.ts`

职责：

- 提供第一版可运行的示例配置。
- 包含主角、塔、敌人三个单位配置。
- 包含一个回合、两个波次。
- 敌人从右侧刷出，向左推进。

后续正式项目中，这个文件可以替换为配置表加载器。

## Command 模块

### `BattleCommandType` 与 `BattleCommand`

文件：`battle/command/battle_command.ts`

职责：

- 定义外部输入命令。
- 当前支持：
  - `PLACE_TOWER`：放置塔。
  - `CAST_ABILITY`：释放技能。

命令基础字段：

- `type`：命令类型。
- `tick`：目标执行 tick。
- `sequence`：同 tick 内排序序号。

设计目的：

- 玩家操作不直接修改战斗状态，而是先转成命令。
- 后续做服务器校验时，可以保存命令日志并复盘。

### `BattleCommandQueue`

文件：`battle/command/battle_command_queue.ts`

职责：

- 缓存外部输入命令。
- 在指定 tick 取出可执行命令。
- 按 `tick` 和 `sequence` 做稳定排序。

关键方法：

- `enqueue()`：加入命令。
- `dequeueForTick()`：取出当前 tick 可执行的命令。
- `getAllCommands()`：查看队列中剩余命令。

## Event 模块

### `BattleEventType` 与 `BattleEvent`

文件：`battle/event/battle_event.ts`

职责：

- 定义战斗逻辑向外输出的事实事件。
- 当前包含单位出生、移动、受伤、死亡，投射物出生、移动、命中，波次/回合开始结束，战斗结束等事件。

设计原则：

- 事件表达“已经发生的逻辑事实”。
- 事件不表达具体表现细节，例如用什么特效、播放什么动画、飘字样式等。
- View 层根据事件自行决定如何显示。

### `BattleEventBus`

文件：`battle/event/battle_event_bus.ts`

职责：

- 收集本 tick 产生的战斗事件。
- `emit()` 写入事件。
- `drain()` 取出并清空事件。
- `peek()` 只查看不清空。

使用方式：

- 逻辑层产生事件时调用 `emit()`。
- `BattleCore.step()` 最后通过 `drain()` 返回事件给外部。

## Unit 模块

### `BattleUnit`

文件：`battle/unit/battle_unit.ts`

职责：

- 表示会影响战斗结果的逻辑单位。
- 当前主角、敌人、塔都使用这个类。
- 保存单位 ID、配置 ID、单位类型、阵营、属性、位置、生命值、下次攻击 tick。

关键方法：

- `isAlive`：是否存活。
- `moveBy()`：移动。
- `takeDamage()`：扣血。

设计说明：

- 这里不是 Cocos 节点，也不是表现对象。
- 它只表达战斗逻辑状态。

### `UnitStats`

文件：`battle/unit/unit_stats.ts`

职责：

- 将 `UnitStatsConfig` 包装成运行时单位属性。
- 当前属性包括最大生命、攻击、射程、攻击间隔、移动速度、投射物速度。

后续扩展：

- 可以加入暴击、攻速倍率、伤害类型、护甲、抗性等。
- 如果 Buff 会修改属性，可以在这里增加基础属性和最终属性的计算方式。

### `UnitController`

文件：`battle/unit/unit_controller.ts`

职责：

- 处理单位行为。
- 当前版本主要处理敌人从右往左移动。
- 敌人走到最左侧时，会让主角死亡，触发失败。
- 提供 `canAttack()` 判断玩家阵营单位是否可以攻击目标。

后续扩展：

- 可以拆出 `EnemyController`、`TowerController`、`HeroController`。
- 可以加入更复杂的 AI、移动规则、目标优先级。

## Ability 模块

### `Ability`

文件：`battle/ability/ability.ts`

职责：

- 表示一个技能逻辑。
- 当前版本的技能会寻找目标，并生成一个投射物。
- 如果命令指定 `targetUnitId`，优先攻击指定目标；否则选择最近敌人。

后续扩展：

- 可以加入释放前摇、冷却、消耗、范围、连发、穿透、分裂等。
- 可以按技能类型派生不同 Ability，或者用配置驱动不同效果。

### `AbilityBook`

文件：`battle/ability/ability_book.ts`

职责：

- 管理可释放技能。
- 当前默认注册 `basic_projectile`。
- 根据 `abilityId` 找到对应 `Ability` 并执行。

设计说明：

- 第一版是全局技能书。
- 后续可以改成每个单位持有自己的 `AbilityBook`。

### `AbilityContext`

文件：`battle/ability/ability_context.ts`

职责：

- 描述一次技能释放上下文。
- 包含释放者、技能 ID、目标单位或目标位置。

### `Projectile`

文件：`battle/ability/projectile.ts`

职责：

- 表示投射物逻辑对象。
- 保存投射物 ID、来源单位、目标单位、伤害、速度、位置、存活状态。

设计说明：

- 投射物会影响战斗结果，所以属于逻辑层对象。
- 投射物表现由 `BattleViewAdapter` 根据事件创建，不在 `Projectile` 内处理。

## Wave 模块

### `WaveScheduler`

文件：`battle/wave/wave_scheduler.ts`

职责：

- 管理回合和波次推进。
- 当前规则：
  - 当前波未开始时，启动当前波。
  - 按 `SpawnConfig` 的 `startTick`、`intervalTicks` 和 `count` 刷怪。
  - 当前波刷完且场上敌人清空后，结束当前波。
  - 当前回合所有波结束后，进入下一回合。
  - 所有回合结束后标记完成。
- 发出回合开始/结束、波次开始/结束事件。

### `RoundController`

文件：`battle/wave/round_controller.ts`

职责：

- 保存当前回合索引和波次索引。
- 提供进入下一波、下一回合的方法。

### `SpawnPlan`

文件：`battle/wave/spawn_plan.ts`

职责：

- 表示运行时刷怪进度。
- 包含原始 `SpawnConfig` 和已刷数量。

## Resolver 模块

### `TargetSelector`

文件：`battle/resolver/target_selector.ts`

职责：

- 负责目标筛选。
- 当前实现是选择攻击者射程内最近的敌方单位。
- 提供距离计算方法。

后续扩展：

- 可增加最近、最远、血量最低、最靠左、Boss 优先、随机目标等策略。

### `DamageResolver`

文件：`battle/resolver/damage_resolver.ts`

职责：

- 负责伤害结算。
- 当前版本直接按传入伤害扣血。
- 返回最终伤害和是否击杀。

后续扩展：

- 暴击、护盾、减伤、伤害类型、元素克制、吸血、反伤等都应优先集中到这里处理。

## Modifier 模块

### `Modifier`

文件：`battle/modifier/modifier.ts`

职责：

- 定义 Buff / Debuff / 光环 / 被动效果的最小结构。
- 当前只包含 ID、所属单位、剩余 tick。

### `ModifierList`

文件：`battle/modifier/modifier_list.ts`

职责：

- 保存和更新一组 Modifier。
- 每 tick 减少持续时间。
- 自动移除过期 Modifier。

当前第一版尚未接入主流程，作为 Buff 系统扩展点预留。

## Random 模块

### `SeededRandom`

文件：`battle/random/seeded_random.ts`

职责：

- 提供带种子的伪随机数。
- 当前使用 xorshift 风格实现。
- 避免战斗逻辑直接使用 `Math.random()`。
- 当前由 `BattleWorld.random` 持有，种子来自 `BattleStartParams.seed`。

当前第一版示例尚未大量使用随机数，但后续如果有随机目标、暴击、随机技能效果，应统一走这里。

## Replay 模块

### `BattleReplay`

文件：`battle/replay/battle_replay.ts`

职责：

- 定义战斗回放/复盘需要保存的数据结构。
- 包含开局参数、命令列表、最终结果。

当前第一版只定义结构，不做服务器复盘。

### `BattleHash`

文件：`battle/replay/battle_hash.ts`

职责：

- 生成调试用状态摘要。
- 当前 `createDebugHash()` 会把 tick 和单位状态拼成字符串。

说明：

- 当前不是正式安全 Hash。
- 后续做服务器校验时，可以替换成稳定 Hash 算法和 Hash 链。

## View 模块

### `BattleViewAdapter`

文件：`battle/view/battle_view_adapter.ts`

职责：

- 逻辑层和 Cocos 表现层之间的适配器。
- 消费 `BattleEvent`。
- 根据单位出生事件创建 `UnitView`。
- 根据单位移动事件更新位置。
- 根据单位死亡事件销毁节点。
- 根据投射物出生、移动、命中事件创建、移动、销毁投射物节点。
- 战斗结束时打印结果日志。

重要约束：

- 不计算伤害。
- 不判断命中。
- 不决定胜负。
- 不修改 `BattleWorld`。

### `UnitView`

文件：`battle/view/unit_view.ts`

职责：

- 表示一个简单 Cocos 表现节点。
- 当前会创建一个 `Node`，加上 `Label` 显示名字，并设置位置。
- 用于展示单位和投射物的最小可视化效果。

后续扩展：

- 单位可以替换为 Prefab、Spine、Animation。
- 投射物可以替换为特效或 Sprite。
- 飘字、命中特效、死亡特效应该独立成新的 View 对象。

## 后续扩展建议

建议按以下顺序继续扩展：

1. 把 `AbilityBook` 从全局改成单位持有，支持不同单位不同技能。
2. 把 `UnitController` 拆成主角、敌人、塔、宠物、队友的控制器。
3. 将 `ModifierList` 接入 `BattleUnit`，让 Buff 能影响属性和行为。
4. 给 `BattleViewAdapter` 增加对象池，避免频繁创建和销毁节点。
5. 增加纯逻辑测试，用 `BattleCore.runUntilFinished()` 验证战斗结果。
6. 等本地闭环稳定后，再补命令日志保存、Replay 和服务器校验。
