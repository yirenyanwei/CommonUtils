# 战斗架构方案

## 目标

本文档描述一个适用于塔防类战斗的战斗架构方案。核心目标是：

- 支持一场游戏包含多个回合，每个回合包含多个波次，波次内按规则产生敌人。
- 支持我方主角、宠物、队友等可扩展战斗单位。
- 战斗逻辑与 View、UI、Cocos 节点完全分离，允许不显示界面只运行逻辑。
- 同一套逻辑可在本地完整运行，后续也可在战斗结束时向服务器提交校验数据。

## 推荐结论

推荐采用“确定性战斗内核 + 命令日志 + 领域对象模型 + 事件输出 + View Adapter”的架构。

这套方案借鉴了几类成熟实践：

- 固定帧战斗模拟：游戏逻辑只按固定 tick 推进，不依赖渲染帧率。
- Command 模式：玩家操作、系统指令、GM 调试指令都转成命令进入战斗内核。
- Event Queue 模式：战斗内核只产出事实事件，外部 View、音效、UI、日志系统自行消费。
- Clean Architecture / Ports and Adapters：战斗核心不依赖 Cocos，不依赖服务器框架，不依赖 UI。
- Actor / Ability / Modifier 模型：常见于 RPG、塔防、卡牌、MOBA 技能系统，适合表达主角、宠物、队友、怪物、技能、Buff、光环等玩法。

不建议第一版采用服务器实时权威同步。当前阶段优先实现“客户端单机完整运行 + 逻辑与 View 分离”，后续再扩展为“战斗结束上传命令日志和结果，服务器复盘校验”。这样实现成本更低，也更适合单机、弱联网和后续反作弊扩展。

## 已确认玩法约束

- 游戏第一阶段按单机战斗实现，完整战斗在本地跑完；服务器校验放到后续阶段，战斗结束时再上报校验数据。
- 地图为自由移动类型，怪物固定从右往左推进。
- 玩家可以放置塔，也可以主动释放技能。
- 失败条件为主角死亡。
- 大部分技能或攻击通过投射物命中结算。
- 第一版只做战斗逻辑与 View 分离，不做服务器复盘。
- 数值计算第一版允许使用浮点数，不要求定点数。

## 架构分层

### 1. Battle Core：纯逻辑战斗内核

职责：

- 保存战斗世界状态。
- 推进固定 tick。
- 处理回合、波次、刷怪、AI、寻路、攻击、技能、伤害、死亡、结算。
- 接收命令，输出事件。
- 生成调试快照，并为后续 Hash 与 Replay 预留扩展点。

约束：

- 不引用 `cc`。
- 不访问 Cocos Node、Prefab、Animation、Audio、UI。
- 不使用 `Date.now()`、`Math.random()`、真实时间、异步计时器参与战斗结果。
- 所有影响战斗结果的随机数必须来自带种子的随机数生成器。
- 所有逻辑通过固定 tick 驱动。

### 2. Battle Data：战斗配置与运行时数据

职责：

- 保存关卡、回合、波次、敌人、角色、宠物、队友、技能、Buff、数值公式等配置。
- 将外部配置转换成战斗内核可读的结构。
- 提供版本号，便于服务器使用同一份配置复盘。

建议将配置分为两类：

- 静态配置：角色表、怪物表、技能表、波次表、关卡表。
- 开局参数：随机种子、玩家阵容、宠物列表、队友列表、局外加成、关卡 ID、配置版本。

### 3. Battle Application：战斗用例层

职责：

- 创建一场战斗。
- 控制战斗开始、暂停、加速、结束。
- 将外部输入转换为战斗命令。
- 将战斗事件分发给表现层、日志系统或服务器上报系统。

这一层可以依赖 Battle Core，但 Battle Core 不依赖这一层。

### 4. Battle View：表现适配层

职责：

- 监听战斗事件。
- 创建、回收、移动、播放 Cocos 节点。
- 播放动画、特效、音效、飘字、UI。
- 根据逻辑实体 ID 维护 View 对象映射。

View 只消费逻辑结果，不参与战斗裁决。比如怪物是否死亡、技能是否命中、伤害是多少，都由 Battle Core 决定，View 只负责显示。

### 5. Server Validator：服务端复盘校验

职责：

- 接收开局参数和命令日志。
- 使用同版本 Battle Core 复盘整场战斗。
- 对关键 tick 或波次结束状态生成 Hash。
- 对比客户端上报的结果、奖励、结算信息。

该模块不进入第一版实现范围。第一版先保证本地逻辑与 View 分离，后续再增加完整复盘、分段校验、随机抽查 tick、Hash 链等反作弊方案。

## 核心设计

### 固定 Tick 驱动

战斗逻辑按固定帧率推进，例如 30 tick/s。

客户端渲染层可以 60 FPS 或更高，但逻辑层不跟随渲染帧率变化。表现层通过插值、补间或直接采样逻辑位置来显示。

建议规则：

- `BattleRunner` 接收真实 delta time。
- `BattleCore` 只接收整数 tick 推进。
- 每个 tick 的执行顺序固定。
- 同一 tick 内的命令、事件、单位遍历顺序必须稳定。

推荐 tick 流程：

1. 读取并排序当前 tick 的命令。
2. 更新回合与波次调度。
3. 处理刷怪计划。
4. 更新单位 AI。
5. 更新移动与寻路。
6. 更新技能、攻击、投射物、Buff。
7. 结算伤害、治疗、死亡、召唤、奖励。
8. 派发战斗事件。
9. 生成调试快照；后续接入服务器校验时再生成校验 Hash。

### 命令模型

命令用于描述“想让战斗发生的输入”，例如：

- 开始战斗。
- 释放主角技能。
- 切换目标。
- 使用道具。
- 放置或升级防御单位。
- 选择局内奖励。
- 调试用跳波、杀怪、加速。

命令需要包含：

- 命令类型。
- 目标 tick。
- 发起者 ID。
- 参数。
- 自增序号。

Battle Core 只在指定 tick 处理命令。第一版命令主要服务本地逻辑与 View 分离；后续服务器复盘时，只要开局参数和命令日志一致，就应得到一致结果。

### 事件模型

事件用于描述“战斗已经发生的事实”，例如：

- 单位出生。
- 单位移动。
- 单位受伤。
- 单位死亡。
- 技能开始、命中、结束。
- Buff 添加、刷新、移除。
- 波次开始、波次结束。
- 回合开始、回合结束。
- 战斗胜利、失败、结算。

事件只向外输出，不允许 View 反向修改战斗状态。表现层可以按事件创建节点、播放动画，也可以在无 UI 模式下完全忽略这些事件。

### 领域对象模型

需要避免“巨型 Unit 类”。推荐使用领域对象组合，将单位、技能、Buff、控制器、数值结算等职责拆开，保持逻辑清晰且便于调试。

核心对象建议：

- `BattleWorld`：一场战斗的完整状态容器。
- `BattleDirector`：战斗流程导演，负责回合、波次、胜负。
- `WaveScheduler`：波次调度，按配置生成敌人。
- `BattleUnit`：战斗单位基类，主角、宠物、队友、敌人都属于单位。
- `UnitController`：单位行为控制器，区分玩家控制、AI 控制、召唤物控制。
- `Ability`：技能对象，负责释放条件、阶段、冷却、消耗。
- `Modifier`：Buff / Debuff / 光环 / 被动效果。
- `Projectile`：投射物逻辑对象。
- `DamageResolver`：伤害、治疗、护盾、暴击、减伤等数值结算。
- `TargetSelector`：目标筛选与排序。
- `PathService`：路径查询，不依赖 Cocos 物理。
- `BattleEventBus`：事件收集与派发。
- `BattleCommandQueue`：命令缓存与排序。

这里的“组合”是面向业务的组合。比如 `BattleUnit` 可以持有 `AbilityBook`、`ModifierList`、`UnitStats`、`UnitController`，让战斗单位的扩展点集中在明确的领域对象上。

### 主角、宠物、队友扩展

建议将我方单位统一建模为 `BattleUnit`，再用阵营、单位类型、控制器和技能配置区分：

- 主角：玩家核心单位，通常有主动技能、成长属性、失败条件。
- 宠物：跟随或自主 AI 单位，可以有被动、主动、光环或触发技能。
- 队友：可作为防守单位、召唤单位或剧情单位，可独立行动。
- 防御塔：如果玩法需要，也可以作为特殊单位，或者作为建筑类战斗对象。

这样后续增加“佣兵、召唤物、临时助战、陷阱、建筑”时，不需要改战斗流程，只需要扩展单位类型、控制器和配置。

### 地图与移动

第一版地图按自由移动处理，不做格子占位和迷宫寻路。怪物的基础推进方向固定为从右往左移动，`PathService` 第一版可以简化为“直线推进 + 局部目标偏移”。

建议第一版规则：

- 敌人从右侧刷怪点进入战场，默认目标为左侧或主角方向。
- 塔和主角可以按照射程选择目标。
- 投射物按逻辑速度移动，命中检测由 Battle Core 完成。
- View 只根据逻辑坐标显示位置，不参与碰撞和命中裁决。

### 回合与波次

推荐将战斗流程拆成三层：

- `BattleDirector` 管理整场战斗状态：未开始、进行中、胜利、失败、结束。
- `RoundController` 管理第几个回合，以及回合开始、回合结束条件。
- `WaveScheduler` 管理当前回合下的多个波次，包括刷怪时间、刷怪点、怪物组、间隔、特殊事件。

波次结束条件建议配置化：

- 本波怪物全部死亡。
- 本波刷怪完成且等待指定 tick。
- 到达固定时间。
- Boss 死亡。
- 手动触发下一波。

回合结束条件也建议配置化。按当前需求，第一版失败条件只实现主角死亡：

- 所有波次结束。
- 主角死亡。
- 达成护送、防守、存活、击杀等目标。

## 逻辑与 View 分离方案

### 战斗实体与表现实体分离

Battle Core 内只保存逻辑实体：

- entityId
- unitType
- position
- hp
- stats
- ability state
- modifier state
- ai state

Battle View 内保存表现实体：

- Cocos Node
- Spine / Animation
- 特效节点
- 血条节点
- 飘字节点
- 音效引用

两者通过 `entityId` 关联。

### 表现层更新方式

推荐事件驱动加状态采样混合：

- 出生、死亡、受伤、释放技能等离散行为用事件驱动。
- 位置、朝向、血量等连续状态可以按 tick 采样。

这样既能减少事件数量，也能让表现层更自然地做插值、补间和特效延迟。

### 无 UI 运行

无 UI 模式只需要：

- 创建 Battle Core。
- 输入开局参数。
- 按 tick 推进。
- 输入命令日志。
- 读取最终结果、事件日志或调试快照。

不需要加载场景、Prefab、Cocos 节点或 UI。这样服务端和自动化测试都可以使用同一套逻辑。

## 确定性与服务器校验

### 必须保证确定性的部分

- 随机数：使用自定义种子随机数，禁止战斗逻辑直接使用 `Math.random()`。
- 时间：使用 tick，禁止战斗结果依赖真实时间。
- 排序：同一 tick 内处理多个对象时必须有稳定排序，通常按 `entityId` 或创建序号。
- 数值：第一版允许使用浮点数计算位置、速度、伤害和投射物命中；后续如果服务器复盘出现跨平台一致性问题，再评估定点数。
- 容器遍历：避免依赖无序对象遍历结果影响战斗结算。
- 外部输入：所有影响结果的玩家操作都必须进入命令日志。

### Hash 与 Replay

建议第一版先预留数据结构，但不实现服务器复盘：

- 保存开局参数、命令日志、配置版本、最终结果。
- 保留生成状态 Hash 的接口位置，但可以先返回调试用结果或不接服务器。
- 本地自动化测试可以先用最终结果对比，验证逻辑和 View 分离是否稳定。

建议第二版扩展：

- 每个波次结束生成一次状态 Hash。
- 每隔固定 tick 生成 Hash。
- 使用 Hash 链防止只伪造最终结果。
- 服务器随机抽查部分 tick 状态。
- 支持从任意快照继续模拟，方便定位不同步问题。

### 客户端与服务端职责

客户端职责：

- 本地运行完整 Battle Core。
- 负责即时表现和操作反馈。
- 第一版只在本地保存开局参数、命令日志和结算结果。
- 后续接入服务器后，上报开局参数、命令日志、关键 Hash、结算结果。

服务端职责：

- 第一版不实现服务端复盘。
- 校验开局参数是否合法。
- 使用同版本 Battle Core 复盘。
- 校验命令是否合法，例如冷却、资源、目标、tick 顺序。
- 校验结算奖励是否可信。

## 推荐目录结构

后续生成代码时，建议使用如下结构：

```text
assets/game/scripts/

    core/
      battle_core.ts
      battle_world.ts
      battle_runner.ts
      battle_director.ts
    command/
      battle_command.ts
      battle_command_queue.ts
    event/
      battle_event.ts
      battle_event_bus.ts
    unit/
      battle_unit.ts
      unit_stats.ts
      unit_controller.ts
      hero_controller.ts
      enemy_controller.ts
    ability/
      ability.ts
      ability_book.ts
      ability_context.ts
    modifier/
      modifier.ts
      modifier_list.ts
    wave/
      round_controller.ts
      wave_scheduler.ts
      spawn_plan.ts
    resolver/
      damage_resolver.ts
      target_selector.ts
    random/
      seeded_random.ts
    replay/
      battle_replay.ts
```

其中 `core`、`command`、`event`、`unit`、`ability`、`modifier`、`wave`、`resolver`、`random`、`replay`、`data` 都不应依赖 Cocos。只有 `view` 允许依赖 `cc`。

## 第一阶段落地范围

建议第一阶段不要一次性做完整塔防大系统，而是先做一个最小可验证闭环：

1. 创建纯逻辑 Battle Core。
2. 支持固定 tick 推进。
3. 支持一场战斗包含 N 个回合、每回合 M 个波次。
4. 支持按波次刷怪。
5. 支持主角和敌人。
6. 支持敌人从右往左自由移动。
7. 支持主角或防御单位自动攻击。
8. 支持伤害、死亡、胜负结算。
9. 支持事件输出。
10. 支持无 UI 跑完整场战斗。
11. 支持玩家放置塔和主动释放技能。
12. 支持投射物创建、飞行、命中和伤害结算。
13. 支持保存开局参数、命令日志和最终结算结果。
14. 接入一个简单 Cocos View Adapter 显示逻辑结果。

宠物、队友、复杂技能、Buff、服务器复盘和 Hash 校验可以在第一阶段骨架稳定后逐步加入。

## 风险与规避

### 风险一：逻辑偷偷依赖 Cocos

规避方式：

- Battle Core 目录禁止导入 `cc`。
- 逻辑层只使用普通 TypeScript 类型。
- View Adapter 负责把逻辑坐标转换为 Cocos 坐标。

### 风险二：早期过度抽象

规避方式：

- 先围绕一个真实关卡闭环设计。
- 技能和 Buff 先做接口稳定，不急着做复杂编辑器。
- 只在出现第二、第三个同类需求后再抽象。

### 风险三：确定性被破坏

规避方式：

- 战斗逻辑禁止使用真实时间和默认随机数。
- 单元测试中用同一份输入重复跑多次，对比最终结果或调试快照。
- 所有影响结果的操作进入命令日志。

### 风险四：View 和逻辑事件粒度不匹配

规避方式：

- 战斗事件表达逻辑事实，不表达表现细节。
- 表现层可以自行聚合、延迟或忽略事件。
- 连续状态使用采样，不强制每 tick 都发移动事件。

## 备选方案对比

### 方案 A：传统 OOP 战斗对象直接驱动 View

优点是最快，适合原型。缺点是逻辑和表现容易混在一起，后期很难做服务器复盘和自动化测试。不推荐作为正式架构。

### 方案 B：完整 ECS

优点是数据导向、性能强、适合大量实体。缺点是技能、Buff、塔防流程对 ECS 初期建模成本较高，团队也需要先建立统一的数据拆分和系统调度规范。若后续战场实体规模明显变大，可以再单独评估。

### 方案 C：服务器全权威实时战斗

优点是安全性高。缺点是开发成本、网络延迟处理、服务器压力都更高。对于塔防类战斗，除非是强 PVP 或多人实时协作，否则第一版不建议。

### 方案 D：确定性内核 + 命令日志 + 复盘校验

优点是逻辑清晰、成本可控、易测试、易复盘、适合弱联网和服务器校验。缺点是需要从一开始约束随机数、时间、排序和数值精度。推荐采用。

## 已确认实现范围

进入代码实现前，当前已确认：

1. 第一版是单机本地跑完整场战斗，战斗结束服务器校验放到后续阶段。
2. 地图是自由移动，怪物固定从右往左移动。
3. 玩家可以放置塔，也可以主动释放技能。
4. 失败条件是主角死亡。
5. 大部分攻击和技能通过投射物命中。
6. 第一版只做战斗逻辑与 View 分离，不做服务器复盘。
7. 第一版允许使用浮点数计算，不要求定点数。
