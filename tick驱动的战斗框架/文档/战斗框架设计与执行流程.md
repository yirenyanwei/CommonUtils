# 战斗框架设计与执行流程

本文档用于帮助理解当前战斗框架如何分层、对象如何协作，以及一场战斗从启动到结束的代码执行流程。

## 一句话理解

当前框架是“用例层驱动纯逻辑战斗，逻辑层产出事件，表现层消费事件”的结构。

```text
Cocos 生命周期
  -> BattleSceneController
      -> BattleApplication
          -> BattleCore
              -> BattleDirector
                  -> CommandSystem
                  -> WaveSystem
                  -> UnitSystem
                  -> AutoAttackSystem
                  -> ProjectileSystem
                  -> BattleResultSystem
              -> BattleEventBus
      -> BattleViewAdapter
```

核心原则：

- 战斗结果只由 `BattleCore` 及其纯逻辑模块决定。
- Cocos View 只展示事件，不决定伤害、命中、死亡和胜负。
- 玩家操作先转成 `BattleCommand`，再由固定 tick 流程执行。
- 战斗逻辑按固定 tick 推进，不跟随渲染帧率变化。

## 分层结构

### 1. 测试入口层

代表类：`TestGame`

职责很薄：

- 接 Cocos 的 `onLoad()` 和 `update(deltaTime)`。
- 创建 `BattleSceneController`。
- 启动测试战斗。
- 每帧把 `deltaTime` 转发给 `BattleSceneController`。

它不应该直接创建 `BattleCore`，也不应该直接拼装战斗命令。后续正式项目里，测试入口可以替换成真正的场景脚本，但下层战斗框架不需要变。

### 2. 场景控制层

代表类：`BattleSceneController`

职责：

- 连接 Cocos 场景和战斗用例。
- 创建 `BattleApplication`。
- 创建 `BattleViewAdapter`。
- 启动示例战斗配置。
- 把战斗逻辑产生的事件交给 View 层显示。

这层可以依赖 `cc`，因为它就是场景侧代码。但它仍然不参与战斗裁决。

### 3. 战斗用例层

代表类：`BattleApplication`

职责：

- 对外提供更像业务语义的方法，比如 `startBattle()`、`placeTower()`、`castAbility()`。
- 内部把业务操作转换为 `BattleCommand`。
- 维护命令序号，保证同一个 tick 内命令顺序稳定。
- 持有 `BattleCore` 和 `BattleRunner`。

这一层不依赖 Cocos。以后做无 UI 测试、服务器复盘、录像回放时，也可以直接用它。

### 4. 战斗核心层

代表类：`BattleCore`、`BattleDirector`、`BattleWorld`

职责：

- `BattleCore` 是逻辑入口，负责接收命令、推进 tick、返回事件。
- `BattleDirector` 是每 tick 的系统管理器，按固定顺序执行各个 `BattleSystem`。
- `BattleWorld` 是上下文容器，保存 tick、结果、随机源、事件总线和各类 Manager。

这一层完全不依赖 Cocos，是战斗结果的唯一来源。

### 5. 领域逻辑模块

代表模块：

- `wave`：回合、波次、刷怪。
- `manager`：管理一类战斗对象，例如单位和投射物。
- `unit`：主角、敌人、塔等战斗单位。
- `ability`：技能和投射物。
- `resolver`：目标选择、伤害结算。
- `system`：固定 tick 内依次执行的战斗系统。
- `event`：战斗事件。
- `command`：玩家或系统输入命令。
- `replay`：后续复盘和校验扩展点。

这些模块围绕 `BattleWorld` 协作，但不直接碰 Cocos 节点。

### 6. 表现适配层

代表类：`BattleViewAdapter`、`UnitView`

职责：

- 监听逻辑事件。
- 根据单位出生事件创建节点。
- 根据移动事件更新节点坐标。
- 根据死亡或命中事件销毁节点。

View 层只看事件，不读取或修改战斗裁决逻辑。

## 启动流程

### 1. Cocos 创建测试入口

`TestGame.onLoad()` 被 Cocos 调用。

流程：

```text
TestGame.onLoad()
  -> new BattleSceneController(this.node)
  -> battleSceneController.startDemoBattle()
```

### 2. 场景控制器启动示例战斗

`BattleSceneController.startDemoBattle()` 做三件事：

```text
createSampleBattleConfig()
  -> battleApplication.startBattle(config, startParams)
  -> new BattleViewAdapter(battleRoot)
  -> enqueueDemoCommands()
```

这里的示例命令包括：

- tick 1 放置第一个塔。
- tick 1 放置第二个塔。
- tick 90 让主角释放一次基础投射物技能。

### 3. 应用层创建核心战斗

`BattleApplication.startBattle()` 内部创建：

```text
BattleCore
BattleRunner
```

`BattleCore` 构造时会：

```text
new BattleWorld(config)
world.unitManager.spawnUnit(heroConfigId, heroPosition)
```

也就是说，主角是在战斗创建时进入 `BattleWorld` 的。

## 每帧执行流程

Cocos 每帧调用：

```text
TestGame.update(deltaTime)
  -> BattleSceneController.update(deltaTime)
```

然后进入：

```text
BattleSceneController.update(deltaTime)
  -> events = BattleApplication.update(deltaTime)
  -> BattleViewAdapter.applyEvents(events)
```

这里要注意：Cocos 是每渲染帧调用一次，但战斗逻辑不一定每渲染帧只走一次 tick。`BattleRunner` 会按固定帧率累计时间，可能一帧推进 0 个、1 个或多个逻辑 tick。

## 每 tick 执行流程

真正的战斗逻辑从 `BattleCore.step()` 开始。

```text
BattleCore.step()
  -> commandQueue.dequeueForTick(world.tick)
  -> director.update(world, commands)
  -> world.tick += 1
  -> eventBus.drain()
```

### 1. 取出当前 tick 的命令

`BattleCommandQueue.dequeueForTick()` 会取出 `command.tick <= world.tick` 的命令，并按：

```text
tick -> sequence
```

排序。这样同一个 tick 内的多个操作顺序是稳定的。

### 2. BattleDirector 调度系统

`BattleDirector.update()` 是每 tick 的系统管线入口。它自身不再承载具体战斗规则，只负责创建上下文并按顺序执行系统：

```text
CommandSystem.update()
WaveSystem.update()
UnitSystem.update()
AutoAttackSystem.update()
ProjectileSystem.update()
BattleResultSystem.update()
```

这个顺序很重要：

- 先执行玩家命令，确保本 tick 放塔或释放技能能尽早生效。
- 再推进波次，刷出当前 tick 应该出现的敌人。
- 再更新单位行为，比如敌人向左移动。
- 再让塔等单位自动攻击。
- 再更新投射物移动和命中。
- 最后检查主角是否死亡、是否胜利或超时。

后续新增 Buff、光环、陷阱、召唤物等逻辑时，优先新增一个系统并插入这条管线，而不是继续把逻辑堆进 `BattleDirector`。

### 3. 命令执行

当前有两类命令：

- `PLACE_TOWER`：调用 `world.unitManager.spawnUnit()` 创建塔。
- `CAST_ABILITY`：调用 `AbilityBook.cast()` 释放技能。

命令不会直接操作 View。比如放塔只会创建逻辑单位并发出 `UNIT_SPAWNED` 事件。

### 4. 波次刷怪

`WaveScheduler.update()` 会：

```text
如果没有激活波次 -> startCurrentWave()
按 startTick / intervalTicks / count 刷怪
如果本波刷完且敌人清空 -> finishCurrentWave()
```

刷怪也是调用 `world.unitManager.spawnUnit()`，因此 View 层仍然只通过事件知道怪物出现。

### 5. 单位更新

`UnitSystem.update()` 会按单位 ID 排序后更新单位。

当前 `UnitController` 主要处理敌人：

```text
enemy.position.x -= moveSpeed
```

敌人到达最左侧时，会直接让主角死亡，随后胜负检查会判定失败。

### 6. 自动攻击

`AutoAttackSystem.update()` 当前只让玩家阵营且不是主角的单位自动攻击，也就是塔。

流程：

```text
TargetSelector.selectNearestEnemy()
UnitController.canAttack()
world.projectileManager.spawnProjectile()
更新 nextAttackTick
```

自动攻击不会立刻扣血，而是先创建投射物。

### 7. 投射物更新和命中

`ProjectileSystem.update()` 会调用 `ProjectileManager.update()` 遍历投射物。

每个投射物：

```text
找到目标单位
向目标移动
如果距离 <= projectile.speed
  -> 发出 PROJECTILE_HIT
  -> DamageResolver.applyDamage()
  -> markUnitDamaged()
  -> 必要时发出 UNIT_DIED
否则
  -> 发出 PROJECTILE_MOVED
```

这就是“投射物命中后才造成伤害”的核心链路。

### 8. 胜负检查

`BattleResultSystem.update()` 当前有三种结果：

- 主角死亡：失败，原因 `HERO_DEAD`。
- 超过最大 tick：失败，原因 `MAX_TICK_REACHED`。
- 所有波次完成且敌人清空：胜利，原因 `ALL_WAVES_CLEARED`。

战斗结束时，`BattleWorld.finish()` 会发出 `BATTLE_FINISHED` 事件。

### 9. 事件交给 View

`BattleCore.step()` 最后调用：

```text
world.eventBus.drain()
```

这会把本 tick 产生的事件返回给上层。

随后：

```text
BattleSceneController
  -> BattleViewAdapter.applyEvents(events)
```

View Adapter 根据事件创建、移动或销毁节点。

## 数据流和控制流

### 玩家操作的数据流

```text
UI / 测试入口
  -> BattleApplication.placeTower() / castAbility()
  -> BattleCommand
  -> BattleCommandQueue
  -> BattleCore.step()
  -> CommandSystem.update()
  -> UnitManager / ProjectileManager 状态变化
  -> BattleEvent
  -> BattleViewAdapter
```

### 刷怪的数据流

```text
BattleConfig.rounds
  -> WaveScheduler
  -> UnitManager.spawnUnit()
  -> UNIT_SPAWNED
  -> BattleViewAdapter 创建 UnitView
```

### 投射物命中的数据流

```text
Tower / Ability
  -> ProjectileManager.spawnProjectile()
  -> PROJECTILE_SPAWNED
  -> ProjectileManager.update()
  -> PROJECTILE_MOVED / PROJECTILE_HIT
  -> DamageResolver.applyDamage()
  -> UNIT_DAMAGED / UNIT_DIED
  -> BattleViewAdapter 更新表现
```

## 为什么这样设计

### 逻辑和表现分开

逻辑层不创建 Cocos 节点，也不播放动画。这样以后可以：

- 不开 UI 直接跑战斗测试。
- 复用逻辑做服务器校验。
- 替换表现层，不影响战斗结果。

### 命令驱动

玩家操作先转成命令，而不是直接改世界状态。这样以后可以：

- 保存命令日志。
- 做 Replay。
- 做服务器复盘。
- 更容易定位“某个 tick 发生了什么”。

### 固定 tick

战斗逻辑用固定 tick，而不是直接用渲染帧 delta。这样：

- 同样输入更容易得到同样结果。
- 攻击间隔、刷怪时间、技能冷却更稳定。
- 后续接服务器校验成本更低。

### 事件驱动表现

逻辑层只发出事实事件，例如“单位出生”“投射物命中”“单位死亡”。View 层自己决定怎么展示。

这能避免逻辑层越来越多 UI、动画、音效依赖。

## 阅读代码建议

建议按这个顺序读：

1. `test_game.ts`：看 Cocos 入口有多薄。
2. `battle/application/battle_scene_controller.ts`：看场景如何启动战斗。
3. `battle/application/battle_application.ts`：看外部操作如何变成命令。
4. `battle/core/battle_core.ts`：看一个 tick 如何进入核心。
5. `battle/core/battle_director.ts`：看每 tick 的完整逻辑顺序。
6. `battle/core/battle_world.ts`：看单位、投射物、事件如何保存和变化。
7. `battle/wave/wave_scheduler.ts`：看回合波次如何刷怪。
8. `battle/view/battle_view_adapter.ts`：看逻辑事件如何变成 Cocos 节点。
