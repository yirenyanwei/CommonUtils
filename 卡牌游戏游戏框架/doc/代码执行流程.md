# 回合制卡牌战斗 · 代码执行流程

> 目的：按"调用链"顺序带你过一遍代码，熟悉每个文件在什么时候被调用、做了什么。
> 建议对照源码阅读，路径均相对 `assets/卡牌游戏/scripts/`。

---

## 0. 总览（一句话）

战斗有两种驱动方式，逻辑内核同一套：

```
A. headless 一次性跑完（服务器校验/回放）：
   runBattle / runBattleWithCommands → 一次产出全部 BattleFrame[]

B. 交互式分步推进（客户端手动技能，当前入口用的就是这种）：
   begin → [stepBeginRound → 玩家可点技能 → stepResolveRound] 循环 → finish
                                                  ↓ 每步产出的帧
                                       BattlePlayer 逐段回放成动画
```

逻辑层（`battle-core/`）只算不画，表现层（`battle-view/`）只画不算，中间靠"战斗记录帧"传递。
玩家主动技能通过**指令队列**喂给逻辑层，并记入 `commandLog`，所以 `seed + 指令序列` 可完整回放/校验。

---

## 1. 入口阶段：BattleEntry（交互式）

文件：`battle-view/battle-entry.ts`

```
start() → runInteractive()
 ├─ battle = createBattle(SAMPLE_CONFIG, SAMPLE_SETUP)   // 装配（还没开打）
 ├─ battle.begin()                          // ① 记 BATTLE_START（含初始快照）
 ├─ buildHeroViews(snapshot)                // ② 据快照建武将节点
 ├─ new PlayerSkillBar(...)                 // ③ 底部主动技能按钮 + 能量条
 ├─ 给每个 HeroView 绑 tap 回调（目标选择用）
 └─ while (!battle.isEnded):                // ④ 逐回合推进
       ├─ frames = battle.stepBeginRound()  //    回合开始+结算已排队的玩家指令
       ├─ player.play(frames)               //    播这一段（期间玩家可点技能→入队）
       ├─ {frames} = battle.stepResolveRound() // 本回合武将出手
       └─ player.play(frames)               //    播这一段
    finish() → player.play(end)             // ⑤ 结果帧
```

关键点：和"一次性算完"不同，这里**逻辑与表现交错**——每推进一步算一小段帧、立刻播出来，期间接收玩家点击。玩家点击只是把指令排队（`enqueuePlayerSkill`），真正生效在下一次 `stepBeginRound`。

> 仍想要"一次性算完"的纯回放/服务器校验，用 `runBattle` / `runBattleWithCommands`（见第 2、7 节）。

## 2. 逻辑层装配：runBattle → createBattle

文件：`battle-core/flow/battle-factory.ts`

```
runBattle(config, setup)
 └─ createBattle(config, setup)
      ├─ instanceCounter = 0                          // 实例 id 计数清零（保证确定性）
      ├─ 构建 allyTeam / enemyTeam
      │    └─ buildHero(config, member, camp)         // 按配置造 Hero，挂上 Skill 列表
      ├─ new BattleContext(config, 两队, rng, events, effects, recorder)
      │    ├─ rng = new SeededRandom(setup.seed)      // 确定性随机（同种子同结果）
      │    ├─ events = new BattleEventBus()           // 事件总线
      │    ├─ effects = createEffectRegistry()        // 注册原子效果处理器
      │    │     register(DamageEffect / HealEffect / AddBuffEffect)
      │    └─ recorder = new BattleRecorder()         // 战斗记录收集器
      ├─ ctx.playerMaxEnergy / playerEnergy 初始化
      ├─ playerSkills = setup.playerActiveSkillIds.map(... new PlayerSkill)
      └─ return new Battle(ctx, playerSkills, energyPerRound)
 └─ battle.run() → { result, frames: battle.frames }
```

记忆点：
- **`SeededRandom`** 是确定性的根，所有"随机"都从它来。
- **`EffectRegistry`** 是扩展点：新增效果类型就在 `createEffectRegistry` 里 `register`。

## 3. 战斗主流程：Battle

文件：`battle-core/flow/battle.ts`

`Battle` 把一场战斗拆成可分步调用的方法，headless 的 `run()` 也是用它们拼起来的：

```
begin()              // 记 BATTLE_START + 快照
stepBeginRound()     // 见 3.1：round++、能量恢复、结算排队的玩家指令
stepResolveRound()   // 见 3.2/3.3：速度排序 → 逐个 takeTurn → 判胜负
finish()             // 记 BATTLE_END + result

// headless 一次性跑完（等价于把上面循环跑到底）：
run(replayCommands?) // 在每个 stepBeginRound 前注入"该回合生效"的预设指令
```

### 3.1 stepBeginRound() — 回合开始 & 结算玩家指令

```
stepBeginRound()
 ├─ round++
 ├─ recorder.add(ROUND_START); events.emit(ROUND_START)
 ├─ playerEnergy = min(max, playerEnergy + playerEnergyPerRound)   // 能量每回合恢复
 └─ commitPendingCommands()                    // 结算"排队中"的玩家主动技能
      └─ 对每条指令：能量够 → 扣能 + PlayerSkill.cast(targetId) + 写入 commandLog
```

要点：
- 玩家点按钮 = `enqueuePlayerSkill()` 只是**排队**（不立即生效、不立即扣能）。
- 真正释放在**下一次 stepBeginRound** 的 `commitPendingCommands`，时机是确定的逻辑坐标（第几回合），所以可回放。
- `PlayerSkill.cast()`（`command/player-skill.ts`）：记 `PLAYER_SKILL_CAST` 帧 → 选目标（玩家指定的 targetId 优先）→ 跑效果。能量扣减由 `Battle` 统一负责。

### 3.2 buildActionOrder() — 速度排序（需求③）

```
所有存活武将，按以下优先级排序：
  1) 速度高的先手 (speed 降序)
  2) 速度相同：我方优先 (camp)
  3) 仍相同：站位靠前优先 (position 升序)
```
排序规则完全确定，无随机，保证可重放。

### 3.3 takeTurn(hero) — 单个武将的一次出手

```
takeTurn(hero)
 ├─ recorder.add(TURN_START); events.emit(TURN_START)
 ├─ ctx.processBuffTriggers(hero, TURN_START)   // 中毒等"回合开始触发"的Buff在此结算掉血
 │     └─ 若被毒死，直接 return
 ├─ if (!hero.buffManager.preventAction)        // 眩晕等控制类Buff会跳过行动
 │     └─ performAction(hero)                    // 见 3.3.1
 ├─ events.emit(TURN_END)
 └─ tickHeroBuffs(hero)                          // Buff 剩余回合 -1，过期则记 BUFF_REMOVE
```

#### 3.3.1 performAction(hero) — 出手决策

```
performAction(hero)
 ├─ 找到"能量够的大招" → 有则:
 │     castSkill(hero, ultimate); hero.energy = 0; recordEnergy
 └─ 否则用普攻:
       castSkill(hero, normal); hero.addEnergy(普攻回能); recordEnergy
```

#### 3.3.2 castSkill(hero, skill) — 释放技能

```
castSkill(hero, skill)
 ├─ recorder.add(SKILL_CAST)
 ├─ events.emit(BEFORE_ATTACK)
 ├─ targets = ctx.selectTargets(hero, skill.targetType)   // 选目标
 └─ ctx.runEffects(hero, skill.effects, targets, skill)    // 跑技能里的原子效果
```

### 3.4 checkEnd() — 胜负判定（需求③）

```
敌方全灭 → ALLY_WIN
我方全灭 → ENEMY_WIN
两边全灭 → DRAW
否则     → null（继续打）
```

## 4. 效果结算：技能如何真正生效

### 4.1 BattleContext.runEffects()

文件：`battle-core/core/battle-context.ts`

```
runEffects(caster, effects[], defaultTargets, skill)
 └─ 对每个 effectConfig:
      ├─ 目标 = effectConfig.targetType 有则单独选，否则用技能的 defaultTargets
      └─ effects.execute({ battle, caster, targets, config, skill })
```

### 4.2 EffectRegistry.execute() → 具体 Effect

文件：`battle-core/effect/effect-registry.ts`

```
execute(context)
 └─ handlers.get(context.config.type).execute(context)   // 按 EffectType 找处理器
```

三个内置效果（`battle-core/effect/`）：

| Effect | 做什么 | 最终调用 |
|--------|--------|----------|
| `DamageEffect` | 算"基础伤害"(攻击×倍率+固定值)后投入伤害管线 | `battle.dealDamage()` |
| `HealEffect` | 算治疗量 | `battle.applyHeal()` |
| `AddBuffEffect` | 给目标加 Buff | `battle.applyBuff()` |

> 注意：Effect 只负责"算基础数值"，暴击/护盾/增减伤/吸血/反伤/死亡等都交给管线与 `BattleContext`，职责清晰。

### 4.3 伤害管线（`combat/damage-pipeline.ts`）

```
dealDamage(ctx, req)            // req: { caster, target, baseAmount, canCrit, trueDamage? }
 ├─ 暴击：BASE_CRIT + 攻方 critChanceBonus → 命中则 ×1.5
 ├─ (非真伤) 攻方增伤 ×(1+damageDealtPercent)
 ├─ (非真伤) − target.def
 ├─ (非真伤) 受方易伤/减伤 ×(1+damageTakenPercent)
 ├─ emit(BEFORE_DAMAGE)
 ├─ 护盾吸收：target.buffManager.consumeShield(amount) → absorbed
 ├─ target.hp -= (amount − absorbed)
 ├─ recorder.add(DAMAGE 帧: value/absorbed/isCrit/hpAfter)
 ├─ emit(AFTER_DAMAGE)
 ├─ 命中后：攻方吸血(lifesteal→applyHeal) / 受方反伤(reflect→真伤回打)
 └─ 若致死：免死(preventDeath→留1血) 否则 ctx.reportDeath()

applyHeal(...)  → ×(1+healingTakenPercent) → 改血 + HEAL 帧 + emit(AFTER_HEAL)
applyBuff(...)  → target.buffManager.add() + BUFF_ADD 帧 + emit(BUFF_ADDED)
```

各阶段修正全部从 Buff 字段聚合（`BuffManager.sumDamageDealtPercent / sumDamageTakenPercent / consumeShield / ...`）。
**加护盾/减伤/吸血/反伤/免死 = 在 Buff 配字段**，不动管线、不动技能。

## 5. Buff 机制（事件驱动的触发器）

文件：`battle-core/buff/`

- **属性类 Buff**（破甲、攻击提升）：在 `BuffConfig.attributeModifiers` 配置。
  Hero 取属性时 `Hero.getAttr()` → `BuffManager.getModifier()` 实时聚合，公式：
  `有效值 = floor(基础值 × (1 + 百分比修正) + 固定修正)`
- **触发类 Buff**（中毒）：配 `triggerTiming + triggerEffects`，
  在 `Battle.takeTurn` 里通过 `ctx.processBuffTriggers(hero, TURN_START)` 触发。
- **控制类 Buff**（眩晕）：配 `preventAction: true`，`takeTurn` 检测到就跳过行动。
- **回合衰减**：`takeTurn` 末尾 `tickHeroBuffs` → `BuffManager.tickDuration()` 回合数−1，到期移除。

## 6. 回放阶段：BattlePlayer.play(frames)

文件：`battle-view/battle-player.ts`

```
play(frames)
 └─ for frame of frames: await playFrame(frame)
      └─ switch(frame.type):
           ROUND_START        → 更新回合label，停顿
           SKILL_CAST         → 施法者 playCast() 抖一下
           PLAYER_SKILL_CAST  → 顶部提示主动技名
           DAMAGE             → 目标 playHurt()闪烁 + setHp() + 飘红字(暴击金字)
           HEAL               → setHp() + 飘绿字
           ENERGY_CHANGE      → setEnergy() 更新武将能量条（无停顿）
           BUFF_ADD/REMOVE    → 更新该武将Buff文字 + 飘字
           DEATH              → die() 半透明
           BATTLE_END         → 显示"胜利/失败"
```

- **表现数据驱动**：`SKILL_CAST/DAMAGE` 不再硬编码特效，而是 `getSkillView(skillId)` 查 `data/skill-view-config.ts` 决定施法动作(冲锋/缩放/抖动)、命中颜色、是否震屏、技能名横幅、音效。**加技能特效 = 加一条配置，不改播放器**。
- 停顿用 `host.scheduleOnce` 包成 `Promise`（`sleep`），形成"一帧一帧播"的节奏。
- 交互模式下 `play(frames)` 会被分段多次调用（每回合两段），同一个 `BattlePlayer` 实例复用，Buff 展示状态跨段保留。
- 表现节点 `HeroView`（`battle-view/hero-view.ts`）纯代码画出：身体方块、名字、血条、能量条、Buff文字，提供 `setHp / playHurt / playCast / die` 以及目标选择用的 `setTapHandler / setSelectable`。
- 主动技能按钮栏 `battle-view/player-skill-bar.ts`：底部能量条 + 每个主动技能按钮（可点/置灰/已排入三态）。
- `battle-entry.ts` 负责把按钮点击转成 `enqueuePlayerSkill`：单体技能先进入"目标选择"（高亮可选武将，点击武将确定目标），群体/自身技能直接入队。
- 飘字在 `battle-view/floating-text.ts`，向上飘动并淡出。

## 7. 确定性 & 服务器校验（需求④）

- 战斗 = `初始配置 + seed + 玩家指令序列` 的纯函数：相同输入 → **逐帧完全相同**的 `BattleFrame[]`。
- 玩家点击不记真实时间，而是记成 `commandLog`（`{round, skillId, targetId}`）——确定的逻辑坐标。
- 校验/回放：`runBattleWithCommands(config, setup, commandLog)` 用**同一份 `battle-core`** 重跑，比对结果即可防作弊。逻辑层零 `cc` 依赖，能直接在 Node 跑。
- 已验证：
  - 无操作同 seed 跑两次，`frames` 完全相等；
  - 交互式分步推进得到的 `commandLog`，喂给 headless 重跑，**逐帧完全一致**。

## 附：一次"普攻打出伤害"的完整调用链

```
Battle.takeTurn
 → performAction
   → castSkill
     → recorder.add(SKILL_CAST)
     → ctx.selectTargets            (target-selector.ts 选到前排敌人)
     → ctx.runEffects
       → EffectRegistry.execute
         → DamageEffect.execute     (算: atk×倍率 − def, 判暴击)
           → ctx.applyDamage
             → emit(BEFORE_DAMAGE)
             → target.hp -= dmg
             → recorder.add(DAMAGE) ← 这帧最终被 BattlePlayer 播成"飘血字+血条下降"
             → emit(AFTER_DAMAGE)
             → (致死) handleDeath → recorder.add(DEATH)
 → ctx.recordEnergy                 (普攻回能 → ENERGY_CHANGE 帧)
```
